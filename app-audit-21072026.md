# Báo cáo audit toàn bộ OpsHub

> Ngày thực hiện: 12/07/2026
> Phạm vi: Flutter Android/Windows/Web, NestJS API, Go realtime, dữ liệu và hạ tầng triển khai production/staging
> Trạng thái: snapshot audit gốc ngày 12/07 được giữ nguyên; remediation và
> bằng chứng runtime cuối được cập nhật đến 15/07/2026 tại mục 12–17

## 1. Kết luận điều hành

OpsHub có nền tảng kỹ thuật tốt ở nhiều lớp: API dùng Helmet, CORS production
có allowlist, DTO bật whitelist, JWT của API chính kiểm tra phiên và token
version, password/reset token được xử lý an toàn, updater kiểm SHA-256/chữ ký và
GitHub Actions ghim action theo commit SHA.

Snapshot ngày 12/07 phát hiện bốn chuỗi Critical: HTTP không redirect, JWT
WebSocket trong query/log, media legacy công khai và break-glass admin hard-code.
Đến 15/07, cùng release `0a949268865afcfb120c8b521bd2c1733ca7ea3a` đã
qua staging và production; HTTPS/HSTS/CSP, one-time realtime ticket,
session-aware authorization, break-glass removal, container/Redis/log/backup
hardening đều có runtime proof. Ba Critical về HTTPS, realtime và break-glass
đã đóng. SEC-03 media legacy đã giảm rủi ro cho dữ liệu mới nhưng vẫn
**partial** cho dữ liệu cũ.

Rủi ro lớn còn mở là **cutover dữ liệu media legacy**: production còn 764 file
trong `/uploads` (~1,5 GB), dry-run tìm 750 reference hợp lệ cần migrate, trong
khi Caddy chưa bật access log nên không thể chứng minh traffic lịch sử bằng số
hit. Vì vậy route public chưa được đóng, migration chưa `--apply` và dữ liệu
chưa bị xóa. Các gate governance còn lại gồm MFA/recovery cho hai admin cá nhân,
credential rotation, retention/ZFS snapshot, live load soak và waiver certificate
Windows tự quản lý đến 13/10/2026.

Tổng hợp mức độ ưu tiên:

| Mức | Số phát hiện | Ý nghĩa |

| High | 8 | Khả năng lạm dụng đáng kể hoặc gây gián đoạn; xử lý trong đợt hardening đầu |

Kết luận về mã không phục vụ runtime:

- Có 19 tệp Dart không reachable từ lib/main.dart trong production graph. Một phần là mã cũ có thể xóa; một phần là demo/test helper nên chuyển khỏi lib; nhóm l10n cần quyết định hoàn tất hay gỡ bỏ, không nên xóa máy móc.
- Không phát hiện tệp NestJS runtime nào hoàn toàn unreachable từ main.ts, nhưng có các method CRUD cây tổ chức cũ không còn route gọi đến.
- Gói release đang chép gần như toàn bộ repository lên máy chủ. Đây là nguồn lãng phí lớn hơn việc xóa vài tệp: source đa nền tảng, test, docs, mockup, tooling và file cấu hình editor đều được giữ trong mỗi release dù không phục vụ runtime.
- Assets icon khoảng 17,5 MB và fonts khoảng 40,1 MB đang được bundle theo thư mục rộng; khoảng 31 MB font không được khai báo sử dụng.
- Container API mang toàn bộ dev dependencies và chạy bằng root. node_modules trong container khoảng 556 MB chưa nén, trong khi dist chỉ khoảng 3,6 MB.

Các số lượng phát hiện bên dưới là **taxonomy của snapshot gốc**, không phải số
lỗi còn mở sau remediation. Trạng thái cuối và nợ còn lại nằm tại mục 17.

## 2. Checkpoint và ranh giới audit

### 2.1 Checkpoint

- Branch tại thời điểm bắt đầu: main.
- HEAD: 4e1ced4b8ecfce8ea33ff3c1440fdb5e5676a25b.
- Production đang chạy đúng release có SHA trên.
- Worktree đã dirty trước audit ở cụm Sales Report, một migration mới và hai tài liệu product/test. Audit không chỉnh sửa hoặc hoàn tác các thay đổi đó.
- Vì production trùng HEAD nhưng worktree local có thay đổi chưa commit, các kết luận runtime được đối chiếu với HEAD/live; kết quả test local phản ánh worktree dirty hiện tại.

### 2.2 Phạm vi đã kiểm tra

- Mã Flutter trong lib, cấu hình Android/Web/Windows, asset và dependency Pub.
- NestJS trong backend-nest/src, Prisma schema/migration, script runtime, dependency npm.
- Go realtime trong backend-go, dependency Go, mô hình WebSocket và Redis pub/sub.
- Caddy, Docker Compose, Dockerfile, workflow production/staging, backup và tài liệu triển khai.
- HTTP response/header công khai của production và staging.
- Runtime production qua SSH ở chế độ chỉ đọc: release SHA, container metadata, log metadata, dung lượng upload/backup, quyền file, migration status và thống kê bảng dữ liệu.
- Reachability tĩnh, tìm kiếm secret có độ tin cậy cao trong cây Git hiện tại, dependency advisory và test/build baseline.

### 2.3 Việc cố ý không làm

- Không thử khai thác, brute force, đăng nhập bằng credential staging, tải nội dung upload thực tế hoặc thay đổi Cloudflare.
- Không đọc/ghi giá trị secret trong file env, không đưa token, mật khẩu, khóa ký hoặc đường dẫn media thực tế vào báo cáo.
- Không dynamic pentest APK/Windows binary, không kiểm tra thiết bị người dùng và không audit tài khoản của nhà cung cấp bên thứ ba.
- Không sửa code, migration, tài liệu product hiện hữu, hạ tầng hoặc dữ liệu.

## 3. Ma trận phát hiện bảo mật

### SEC-01 — Critical — Web production/staging không ép HTTPS

**Bằng chứng**

- HTTP tới production và staging trả 200 với nội dung trang, không trả redirect.
- HTTPS hoạt động nhưng trang static gốc không có HSTS, CSP, X-Content-Type-Options, X-Frame-Options, Referrer-Policy hoặc Permissions-Policy.
- API động có header Helmet; thiếu sót nằm ở lớp static/edge.
- deploy/home-server/Caddyfile:1 khai báo site bằng HTTP vì chạy sau Cloudflare Tunnel và chưa có bộ security header chung.

**Chuỗi nguyên nhân → tác động**

Người dùng gõ hoặc mở link HTTP → trình duyệt nhận JavaScript qua kênh có thể bị sửa → script giả có thể lấy thông tin đăng nhập hoặc dữ liệu trước khi API CORS chặn origin → rủi ro chiếm tài khoản.

**Đề xuất**

- Bật Always Use HTTPS/HSTS ở Cloudflare và kiểm tra cả production lẫn staging.
- Thêm CSP theo nonce/hash hoặc CSP tối thiểu phù hợp Flutter Web, nosniff, frame-ancestors, Referrer-Policy và Permissions-Policy cho static.
- Chỉ bật HSTS sau khi xác nhận toàn bộ subdomain cần thiết đều hỗ trợ HTTPS.

### SEC-02 — Critical — JWT WebSocket nằm trong query/log và không có revocation theo phiên

**Bằng chứng**

- lib/core/constants/api_constants.dart:136-151 luôn gắn access_token vào URL WebSocket.
- backend-go/main.go cho phép lấy bearer token rồi fallback sang query; middleware gin.Default ghi RawQuery vào access log.
- Log realtime production có 79 dòng chứa access_token trong 30 phút và 297 dòng trong 2 giờ; không trích token vào báo cáo.
- Go chỉ kiểm chữ ký HS256 và exp, không đối chiếu user/session/tokenVersion, issuer hoặc audience như NestJS.
- JWT API hiện có thời hạn dài; logout, đổi mật khẩu hoặc thay phiên trên NestJS không thu hồi tức thời kết nối realtime dùng token cũ.

**Chuỗi nguyên nhân → tác động**

JWT nằm trong URL → proxy/logger/process history có thể lưu lại → người có quyền đọc log dùng lại token trước khi hết hạn → Go vẫn chấp nhận dù phiên API đã bị thu hồi.

**Đề xuất**

- NestJS cấp WebSocket ticket dùng một lần, TTL 30–60 giây, lưu/consume nguyên tử trong Redis và gắn user, scope, client, audience.
- Go log đường dẫn đã redaction query; không dùng gin.Default trực tiếp.
- Chuyển client sang ticket, đo tỷ lệ client cũ rồi tắt access_token query.
- Sau khi đóng đường rò: rotate JWT secret, tăng tokenVersion/thu hồi phiên và xử lý log cũ theo chính sách sự cố.

### SEC-03 — Critical — Upload bảo hành/feedback/avatar đang công khai

**Bằng chứng**

- deploy/home-server/Caddyfile:54-57 dùng file_server công khai cho /uploads/*, không có guard xác thực.
- backend-nest/src/upload/upload.service.ts:17-120 tạo tên/đường dẫn có cấu trúc theo nghiệp vụ; đây không phải authorization boundary.
- Runtime: khoảng 760 tệp, tổng 1.512.075.781 byte; toàn bộ tệp kiểm tra được có quyền đọc bởi mọi user trên host. Thư mục gốc có mode 755.
- Không thấy lifecycle dọn avatar cũ, ảnh bảo hành dư hoặc orphan media.

**Chuỗi nguyên nhân → tác động**

Người dùng upload ảnh có dữ liệu cá nhân/nghiệp vụ → URL static không kiểm quyền → ai có hoặc đoán được URL có thể tải → cache/proxy có thể giữ bản sao ngay cả sau khi UI không còn hiển thị.

**Đề xuất**

- Tách asset công khai của Help khỏi media riêng tư.
- Lưu media riêng bằng opaque object id; tải qua endpoint kiểm JWT + feature + record scope hoặc signed URL rất ngắn hạn.
- Triển khai dual-read và backfill metadata trước, đổi client, sau đó mới đóng Caddy route và purge cache.
- Đặt Cache-Control private/no-store, nosniff; bổ sung retention, xóa orphan và audit access.

### SEC-04 — Critical — Break-glass admin hard-code và tự tái kích hoạt

**Bằng chứng**

- backend-nest/src/auth/break-glass-admin.constants.ts:1-7 hard-code email và bcrypt hash.
- backend-nest/src/user/user.service.ts:398-404 gọi bootstrap khi module khởi tạo.
- backend-nest/src/user/user.service.ts:4988-5034 tạo tài khoản nếu thiếu và ép lại role SUPER_ADMIN, trạng thái active, scope toàn quốc ở mỗi lần khởi động.
- backend-nest/src/auth/auth.service.ts:303-305 cho tài khoản này đi qua ngoại lệ domain.
- Runtime production xác nhận có đúng một tài khoản break-glass đang active.

**Chuỗi nguyên nhân → tác động**

Định danh và password verifier cố định nằm trong source; verifier chỉ được seed khi tạo mới hoặc khi tài khoản chưa có password, còn password hiện hữu được giữ nguyên → xóa tài khoản sẽ làm startup tạo lại bằng verifier cố định, còn disable/demote sẽ bị startup nâng lại → kết hợp với rate limit có thể né và không có MFA → đường vào quản trị có tính lâu dài.

**Đề xuất**

- Loại bootstrap khỏi startup.
- Chuyển break-glass thành quy trình một lần có phê duyệt, secret ngoài Git, TTL ngắn, MFA và audit bắt buộc.
- Xác nhận còn ít nhất hai admin bình thường hoạt động rồi vô hiệu hóa tài khoản hiện tại, rotate mọi secret liên quan và thu hồi phiên.

### SEC-05 — High — Rate limit có thể né, OTP không dùng CSPRNG và public auth lộ trạng thái tài khoản

**Bằng chứng**

- backend-nest/src/common/user-aware-throttler.guard.ts:34-65 chọn một tracker duy nhất; khi không có JWT hợp lệ, tracker có thể lấy từ x-client-id, x-device-id, query hoặc body do caller tự đặt.
- app.module.ts chỉ có global 120 request/phút, chưa có bucket kết hợp IP + account + device theo endpoint.
- password-reset.service.ts:352-361 và email-verification.service.ts:85-87 tạo mã bằng Math.random.
- Login, đăng ký/xác minh và quên mật khẩu trả thông điệp khác nhau khi email có/không tồn tại.

**Tác động**

Kẻ tấn công xoay device/client id để né bucket, dò email hợp lệ và thử OTP/password có chủ đích.

**Đề xuất**

- Dùng nhiều bucket đồng thời: IP prefix, account/email hash, device đã xác minh, endpoint và global.
- Dùng crypto.randomInt cho OTP; TTL ngắn, one-time consume, attempt counter và cooldown.
- Public response thống nhất; chi tiết chỉ ghi log đã redaction.
- Thêm lockout/risk signal và MFA cho quản trị.

### SEC-06 — High — Go realtime thiếu authorization theo event và có điểm nghẽn/DoS

**Bằng chứng**

- canReceive trong backend-go/main.go chỉ scope một số loại payment/statement/offset; event warranty và SalesReport mặc định cho mọi user đã xác thực.
- backend-nest/src/warranty/warranty.service.ts:173-178 publish payload không có audience/store scope.
- Sales Report publish có recipient/store metadata nhưng Flutter tự lọc ở client; client-side filter không phải authorization.
- Hub ghi tuần tự đến mọi connection, chưa có queue riêng, write deadline hoặc cô lập slow client.
- /ws/app-updates là public và chưa có giới hạn connection/handshake/idle rõ ràng.

**Tác động**

User hợp lệ có thể nhận metadata ngoài phạm vi; một client chậm hoặc nhiều connection có thể làm nghẽn broadcast.

**Đề xuất**

- Định nghĩa schema audience bắt buộc cho từng event và lọc tại Go trước khi enqueue.
- Queue có giới hạn theo connection, writer goroutine riêng, deadline và chính sách drop/disconnect.
- Rate limit handshake, cap connection theo IP/user, idle timeout và readiness phụ thuộc Redis.

### SEC-07 — High — Credential SUPER_ADMIN staging được lưu trong repo

**Bằng chứng**

- deploy/staging/README.md chứa mật khẩu staging dạng literal.
- Script refresh dữ liệu tạo tài khoản staging với role SUPER_ADMIN và các tài khoản dùng chung khác.
- Staging có endpoint Internet công khai và cũng đang phục vụ HTTP.

**Tác động**

Ai đọc được repo hoặc lịch sử có thể biết credential dùng chung; staging thường chứa code mới và tích hợp gần production nên có thể trở thành bàn đạp.

**Đề xuất**

- Rotate ngay, bỏ literal khỏi cây hiện tại và dọn Git history theo quy trình có kiểm soát.
- Đặt staging sau Cloudflare Access/VPN/IP allowlist.
- Tạo tài khoản tạm theo người dùng, TTL ngắn, quyền tối thiểu; tắt outbound integration khi refresh dữ liệu.

### SEC-08 — High — CSV formula injection

**Bằng chứng**

- backend-nest/src/map-vietin/map-vietin.service.ts:3768-3799 xuất content/tên người trả và dữ liệu ngoài hệ thống qua helper chỉ quote dấu phẩy/dấu nháy.
- backend-nest/src/offset-adjustments/offset-adjustments.service.ts:988-1008 và helper 1054-1064 xử lý note/rejectReason tương tự.
- Cell bắt đầu bằng =, +, -, @, tab, CR hoặc LF chưa được neutralize.

**Tác động**

Nhân viên mở CSV trong Excel có thể kích hoạt formula do dữ liệu giao dịch/ghi chú điều khiển.

**Đề xuất**

- Ưu tiên XLSX với cell kiểu string.
- Nếu giữ CSV, dùng helper trung tâm chống formula injection và test các ký tự nguy hiểm, kể cả trường hợp delimiter/quote làm thay đổi vị trí đầu cell.

### SEC-09 — High — Dependency có advisory đã biết

**Kết quả tại 12/07/2026**

- npm audit --omit=dev: 5 High, 1 Moderate.
- Multer 2.1.1: advisory DoS liên quan field lồng sâu và cleanup upload bị hủy.
- xlsx 0.18.5: prototype pollution/ReDoS, npm không có bản fix; app dùng để parse file Excel admin/FIFO.
- Nodemailer 8.0.7: nhiều advisory mức High; phần lớn đường khai thác không trùng cách gọi hiện tại nhưng vẫn cần nâng.
- Hono 4.12.18 nằm trong tooling Prisma nhưng bị đưa vào runtime do copy toàn bộ node_modules.
- qs 6.15.0 có advisory mức Moderate.
- govulncheck tìm thấy 13 vulnerability đi qua call graph: 12 ở Go standard library đã có bản vá theo patch release và 1 ở quic-go 0.59.0, fix ở 0.59.1.
- Quét OSV cho 157 package Pub không trả về advisory đã biết; điều này không đồng nghĩa không có lỗ hổng.

**Đề xuất**

- Nâng Multer/Nest platform, Nodemailer, qs và quic-go; ghim Go patch release đã vá.
- Thay xlsx bằng thư viện được duy trì; trước khi thay phải có golden test cho import/export hiện tại.
- Tách dev dependencies khỏi image runtime.

### SEC-10 — High — Upload dùng memory buffer lớn và chỉ tin MIME khai báo

**Bằng chứng**

- backend-nest/src/upload/image-upload.options.ts cho tối đa 20 tệp × 10 MB trong memory.
- Validation hiện dựa trên MIME từ multipart; chưa kiểm magic bytes/decode, re-encode, strip EXIF hoặc scan nội dung.
- Rate limit có thể né như SEC-05.

**Tác động**

Một user đã xác thực có thể gây áp lực bộ nhớ bằng request song song; file giả MIME hoặc metadata nhạy cảm được giữ nguyên và public theo SEC-03.

**Đề xuất**

- Đặt aggregate request cap, stream vào vùng tạm có quota, xác minh magic bytes và decode giới hạn pixel.
- Re-encode định dạng cho phép, strip EXIF, tùy rủi ro bổ sung malware scan.
- Quota theo user/feature/record và cleanup khi request bị hủy.

### SEC-11 — High — API container chạy root, mang dev tooling và log không giới hạn

**Bằng chứng**

- backend-nest/Dockerfile:18-23 copy toàn bộ node_modules, Prisma và scripts từ build stage.
- Container API production chạy uid 0; root filesystem writable, chưa drop capability/no-new-privileges.
- node_modules runtime khoảng 36.722 tệp/556 MB chưa nén; dist khoảng 370 tệp/3,6 MB.
- Docker dùng json-file mà không có max-size/max-file.
- API log khoảng 22,5 MB và 55.159 dòng trong khoảng hai giờ; 396 dòng khớp dạng email.

**Tác động**

Khi có RCE hoặc package compromise, blast radius cao hơn; dev tooling tăng bề mặt; log có thể đầy disk và lưu PII không cần thiết.

**Đề xuất**

- Image runtime chỉ có production dependencies và artifact cần chạy; migration/ops tách image/job riêng.
- USER không phải root, read-only filesystem, tmpfs cần thiết, cap_drop ALL, no-new-privileges.
- Log rotation ở Docker; structured logging, sampling và hash/redact email/identifier.

### SEC-12 — High — Backup thủ công có quyền đọc rộng và không mã hóa

**Bằng chứng**

- /srv/opshub/backups có 11 tệp; 6 tệp có mode cho phép user khác trên host đọc, gồm archive dung lượng lớn.
- deploy/home-server/backup.sh:20-38 tạo gzip database/upload nhưng không đặt umask 077, chmod 600 hoặc mã hóa.
- Audit không truy cập được đích scheduled backup chính, nên chưa kết luận trạng thái toàn bộ chuỗi backup.

**Tác động**

User/process khác trên host hoặc bản sao lưu bị lấy cắp có thể đọc database/media.

**Đề xuất**

- Directory 700, file 600, umask 077; mã hóa trước khi rời host và quản lý key tách biệt.
- Retention, immutable/offline copy và restore drill định kỳ.

**Cập nhật bằng chứng runtime ngày 13/07/2026**

- Đã tạo age identity ngoài production; private identity chỉ nằm trong VeraCrypt
  container trên USB và bản container dự phòng đã qua hai vòng mount/hash. Chỉ
  public recipient được chuyển lên host.
- Đã tạo backup mã hóa tại

### SEC-13 — Medium — Endpoint app-logs có thể bị dùng làm kênh bơm dữ liệu

- POST /app-logs cần JWT nhưng không có feature guard riêng.
- Context là object tùy ý; chưa có giới hạn serialized size, depth, key count/source allowlist ngoài body limit chung.

Đề xuất: limit riêng theo user/app version, tối đa khoảng 16 KB/event, giới hạn depth/key, allowlist source/level, quota ngày và drop debug ở server.

### SEC-14 — Medium — Base URL tích hợp credential chưa fail-closed theo HTTPS/host

- Config validate PUBLIC/eFAST còn cho HTTP; ERP, Map Vietin và TTS base URL chưa có allowlist host/HTTPS rõ ràng.

Đề xuất: production chỉ chấp nhận HTTPS và exact host allowlist, chặn redirect sang host khác, timeout/size cap và egress policy.

### SEC-15 — Medium — Docker local publish database/Redis ra mọi interface

- docker-compose.yml publish PostgreSQL 5432 với credential dev dạng literal và Redis 6379 không auth.

Đề xuất: bind 127.0.0.1, secret qua env, Redis auth hoặc không publish host; tách profile integration test.

### SEC-16 — Medium — Chuỗi ký Android/manual release chưa fail-closed hoàn toàn

- android/app/build.gradle.kts:83-88 fallback release sang debug signing nếu thiếu release signing.
- Workflow production/staging hiện fail-closed và yêu cầu secret, đây là control tốt; rủi ro nằm ở build release thủ công.
- File signing local được ignore và không thấy trong Git history, nhưng ACL có thêm principal ngoài owner có quyền sửa. Báo cáo không ghi giá trị.
- Manifest chưa có allowBackup/dataExtractionRules rõ ràng; app có cache email/org/role trong SharedPreferences.
- Self-updater kiểm hash/chữ ký nhưng Dart còn chấp nhận URL HTTP.

Đề xuất: build production release phải fail nếu thiếu signing; thu hẹp ACL/rotate credential; đặt backup rule; production update URL chỉ HTTPS.

### SEC-17 — Medium — VietQR external API key chấp nhận qua query/body

- Controller cho phép API key từ header, body hoặc query.
- Query/body dễ đi vào access log, history hoặc error capture; so sánh string thường không constant-time.

Đề xuất: chỉ nhận header, dùng timingSafeEqual hoặc HMAC/key id, có rotation và tách quyền đọc/ghi.

### SEC-18 — Medium — Static Help/Download cần URL policy và CSP chặt hơn

- Các trang custom có escape HTML và sanitize URL cơ bản, đây là điểm tốt.
- Help sanitizer vẫn cho protocol-relative URL bắt đầu bằng //; download manifest không ép scheme/host cùng origin.

Đề xuất: exact same-origin HTTPS allowlist, CSP không unsafe-inline nếu có thể, Trusted Types khi phù hợp và test payload độc hại.

### SEC-19 — Medium — Redis/internal network và container policy còn tin cậy môi trường quá mức

- Redis production không publish ra host nhưng không có password và protected-mode tắt trong network nội bộ.
- API/Caddy/Postgres/Redis chưa có hardening đồng đều; image dùng tag nổi thay vì digest.

Đề xuất: ACL/password, network tách theo service, digest pinning, automated update có kiểm thử và policy container chung.

### SEC-20 — Medium — CORS reject bằng lỗi 500 và credentials không cần thiết

- Origin HTTPS production được cho phép; HTTP origin và evil origin trả 500 do callback Error.
- API dùng bearer token, không thấy cookie auth cần credentials=true.

Đề xuất: từ chối origin sạch, không tạo 500/noise; tắt credentials nếu không có cookie use case; thêm test preflight cho prod/staging.
